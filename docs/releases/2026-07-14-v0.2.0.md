## Rollback

Finish or cancel v0.2 jobs, preserve run evidence and changed worktrees, reinstall v0.1.2 through the personal marketplace, and restore the matching global rule. Do not attempt to resume non-terminal state version 2 with v0.1.2.

## Known boundaries

- The local setup page is intentionally one-shot and not a persistent dashboard.
- An uncertain external write blocks for read-only reconciliation rather than inventing a retry decision.
- Model and Fast availability remain dependent on the current Codex catalog and account.
- Codex CLI 0.144.0 forwards the requested tier but does not expose the backend-returned tier in normal exec JSON/session output. Normal reports therefore distinguish `service_tier_request_verified=true` from `service_tier_observation_status=not_exposed`. Automatic Standard retry works when rejection or degradation is observable; a hidden backend downgrade cannot be detected without transport inspection.
- `follow-entry` uses the entry Service Tier only when the current Codex surface exposes it. If only model/reasoning are present, the job records `service_tier_source=default-unavailable` and uses Standard instead of guessing.

## Service Tier compatibility finding

Official Codex source at commit `767822446c7a594caa19609ca435281a9ec67e0d` confirms that a Fast selection serializes as `priority` in the Responses request, while the SSE completion parser does not retain `response.service_tier`. A bounded local ChatGPT-backend smoke capture on 2026-07-14 independently observed a `gpt-5.6-terra` request with `priority` and a completed response with `default`. The sanitized two-field-path evidence is stored in [service-tier-smoke.jsonl](evidence/2026-07-14-service-tier-smoke.jsonl); raw traffic and credentials are not part of the repository.
